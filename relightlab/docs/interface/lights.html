Lights geometry

RTI software only considers the light directions, as if the lights were infinitely distant. This approximation is somewhat ok for a visual inspection, but severely distorts the normals. We can instead take into account for each pixel the real light direction if we can reconstruct the geometry of the scene: 3d positions of the lights and size of the image.

If the images are captured using a dome, you can provide the light directions/positions to relightlab instead of using reflective spheres. You can use an .lp file (see formats/lp.md

Alternatively use a shiny-ball capture and process the image set to generate a .dome file you can reuse as long as the camera and lights do not change.

Focal length (35mm equivalent)

The focal length is expressed as its 35mm equivalent — i.e. normalised to a full-frame (36 × 24 mm) sensor. This is needed to convert pixel coordinates to physical angles and to compute per-pixel light directions when using 3D light positions.

If the value is missing, import it with the Read from images button, which reads the EXIF metadata of the first valid image. You can also type it manually.

Input colorspace

RelightLab always converts images to linear RGB before processing, because RTI and normal-map mathematics require a physically linear light response.

When an embedded ICC profile is present in the JPEG files, the correct conversion is performed automatically and no action is needed.

When no ICC profile is found, RelightLab cannot determine the colorspace of the images:

  • sRGB — choose this for photographs taken with a standard camera or processed in a photo editor that writes sRGB output. This is the default.
  • Linear RGB — choose this for synthetic renders (e.g. Blender) that output raw linear values without gamma encoding.
Choosing the wrong option will produce incorrect normals and RTI relighting.
